# Deployment — Cloudflare Workers Static Assets

Both apps deploy as **assets-only Cloudflare Workers**, built and pushed by
`.github/workflows/deploy.yml` on every push to `master`.

| | app | Worker name | config | production hostnames |
| --- | --- | --- | --- | --- |
| Public site | `apps/website` | `collection-website` | `apps/website/wrangler.jsonc` | `thecollectionisb.com.pk`, `www.thecollectionisb.com.pk`, `thecollectionisb.pk`, `www.thecollectionisb.pk` |
| Operator dashboard | `apps/dashboard` | `collection-dashboard` | `apps/dashboard/wrangler.jsonc` | `admin.thecollectionisb.pk` |

**Two domains, therefore two Cloudflare zones.** The public site is served on **both**
registrations — `thecollectionisb.com.pk` (the **canonical** origin) and `thecollectionisb.pk`, apex
and `www` on each. The operator dashboard is served **only** on `admin.thecollectionisb.pk`. Both
zones must be onboarded to Cloudflare, and the single deploying API token must carry DNS permissions
on **both** — one workflow run deploys both Workers.

Because the site answers on two domains with identical content, every route declares
`thecollectionisb.com.pk` canonical — twice over, once in JS and once as an HTTP header. See step 8;
the invariants there are easy to break by accident.

The dashboard being on a different registrable domain is a small security win as well as a tidy
split: it shares no cookie scope with the public site, and an HSTS `includeSubDomains` policy on one
cannot reach the other. The two Workers have separate route lists, so no website hostname can reach
the dashboard.

## How it works, and the three things that will bite you

**1. `VITE_*` variables are BUILD-time, not runtime.** Vite statically replaces
`import.meta.env.VITE_SUPABASE_URL` / `VITE_SUPABASE_ANON_KEY` into the JS bundle when it compiles
(`packages/shared/supabase.ts` is their only reader in the entire repo). They therefore belong in the
**GitHub Actions build step**, as repository secrets. Setting them as Cloudflare Worker variables or
via `wrangler secret put` does nothing — that is a separate store, read via `env.X` inside a Worker
script, and these Workers *have no script*. The failure mode is silent: a bundle containing
`undefined`, no build error, and a confusing Supabase init failure at page load. The workflow asserts
both secrets are non-empty before building, so this fails loudly instead.

**2. The Workers are assets-only on purpose — never add a `main`.** Cloudflare does **not** apply
`_headers` rules to responses generated by Worker code. With no `main` and no `assets.run_worker_first`,
every response is an asset response, so the dashboard's `X-Robots-Tag: noindex, nofollow` always
applies. Adding an SSR or API Worker in front would silently drop it. (It also keeps every request
free and unmetered.)

**3. `_headers` only works from inside `assets.directory`.** It reaches `dist/` via Vite's verbatim
copy of `public/`. A `_headers` at the repo root or beside `wrangler.jsonc` is ignored without error.
CI asserts `dist/_headers` exists on every build.

### Why CI builds instead of Cloudflare's Git integration

Cloudflare's build image ships **pnpm 10.11.1**. This repo's `pnpm-workspace.yaml` uses `allowBuilds`
(needs pnpm ≥ 10.26.0) and `minimumReleaseAge` (≥ 10.16.0), and pnpm *silently ignores* workspace keys
it doesn't recognise. On that image nothing would be approved to run install scripts, so `esbuild`'s
and `@tailwindcss/oxide`'s postinstall steps would never run and `vite build` would fail — or worse,
emit no CSS without failing. Cloudflare's images also ignore `engines`, don't infer pnpm from the
lockfile, and don't document `packageManager`/corepack support; a dashboard-only `PNPM_VERSION` build
variable is the sole lever there. GitHub Actions pins pnpm `11.0.8` and Node via `.node-version`, in
files that live in the repo.

Do **not** remove `supportedArchitectures.os: [linux]` from `pnpm-workspace.yaml` — it is what puts
the Linux native binaries into the lockfile. The three win32 `optionalDependencies` in the root
`package.json` are skipped silently on Linux and do not trip `--frozen-lockfile`.

## Cutover runbook

Steps are ordered so nothing is wasted if step 0 fails.

### 0. Nameserver delegation — confirmed available

A Workers **Custom Domain requires the hostname's nameservers to be managed by Cloudflare** — stricter
than Vercel and stricter than Cloudflare Pages. Delegation has been confirmed possible for both
registrations, so the Workers path below is viable end to end. **Both** `thecollectionisb.com.pk` and
`thecollectionisb.pk` must be delegated; the dashboard cannot attach until the `.pk` zone is Active.

> Kept for reference: if delegation of either domain later proves impossible, that hostname moves to
> **Cloudflare Pages**, which accepts a plain CNAME from external DNS. The `_headers` files carry over
> unchanged; `wrangler.jsonc` is replaced by Pages project settings and the SPA fallback becomes Pages'
> automatic SPA detection. It changes the deploy commands, so ask first.

### 1. Register the workers.dev subdomain and check the names are free

Cloudflare dashboard → **Workers & Pages → Overview**. Confirm the account has a `*.workers.dev`
subdomain registered and **write it down** — every verification below uses it. If the account has none,
`wrangler deploy` with `workers_dev: true` will try to prompt for one interactively and **hang the CI
job**. Also confirm `collection-website` and `collection-dashboard` are unused in this account.

### 2. Add the GitHub Actions secrets

GitHub repo → Settings → Secrets and variables → Actions. Add exactly four:

| secret | value |
| --- | --- |
| `VITE_SUPABASE_URL` | from your local `apps/*/.env` |
| `VITE_SUPABASE_ANON_KEY` | the **anon/public** key only — never `service_role` |
| `CLOUDFLARE_API_TOKEN` | see below |
| `CLOUDFLARE_ACCOUNT_ID` | Workers & Pages → Overview, right-hand sidebar |

For now the token can use the stock **"Edit Cloudflare Workers"** template — that is enough for
workers.dev deploys. **It will not be enough for step 6**, and Cloudflare fixes a token's zone scope at
creation time, so it gets re-issued once the zone exists.

### 3. Push and let the first deploy run

Push everything **except** the `vercel.json` deletions. Vercel keeps serving live traffic untouched.

In the Actions log confirm: pnpm 11.0.8 is used; `pnpm install --frozen-lockfile` prints no
`Ignored build scripts` warning and does not fail with `ERR_PNPM_IGNORED_BUILDS` or
`ERR_PNPM_OUTDATED_LOCKFILE`; both `dist/_headers` assertions pass; both deploys print a workers.dev URL.

### 4. Verify on workers.dev before touching DNS

```bash
W=https://collection-website.<your-subdomain>.workers.dev
D=https://collection-dashboard.<your-subdomain>.workers.dev

# SPA fallback: every client-side route must be 200 + text/html, not 404.
for p in / /collection /collection/does-not-exist /privacy /terms /nonsense; do
  curl -sI "$W$p" | awk -v p="$p" 'NR==1{s=$2} /^[Cc]ontent-[Tt]ype:/{t=$2} END{print p, s, t}'
done
curl -s "$W/collection/xyz" | grep -q '<div id="root">' && echo "fallback serves the app shell"

# Trailing-slash form must not redirect somewhere react-router mishandles.
curl -sI "$W/collection/" | head -1

# Real files must NOT be shadowed by the fallback.
for f in /robots.txt /sitemap.xml /og-image.jpg /favicon.svg; do
  curl -sI "$W$f" | awk -v f="$f" 'NR==1{s=$2} /^[Cc]ontent-[Tt]ype:/{t=$2} END{print f, s, t}'
done

# Hashed assets immutable; index.html must NOT be.
curl -sI "$W/index.html" | grep -i cache-control
curl -s "$W/" | grep -o '/assets/index-[^"]*\.js' | head -1   # then curl -sI that path

# Dashboard: noindex on EVERY response, including assets and bogus paths.
for p in / /anything /favicon.svg; do
  curl -sI "$D$p" | grep -i x-robots-tag || echo "MISSING on $p"
done

# _headers must not be publicly served. If this returns rule text, add a
# .assetsignore file containing "_headers" and redeploy.
curl -s -o /dev/null -w '%{http_code}\n' "$W/_headers"
```

```bash
# Website on workers.dev: noindex MUST be present here (and absent in step 11).
curl -sI "$W/" | grep -i x-robots-tag || echo "MISSING — the workers.dev copy is indexable"
```

On `*.workers.dev` **both** apps must carry `X-Robots-Tag` — the dashboard on every response, and the
website via the `https://:version.:subdomain.workers.dev/*` rule in `apps/website/public/_headers`,
which stops this hostname competing with the real domain in search results. On the **real** domain
(step 11) the website must **not** carry it, and the dashboard still must, everywhere.

### 5. Check CAA records

Check **both** zones — the dashboard's certificate is issued against a different registration:

```bash
# Public site zone
dig CAA thecollectionisb.com.pk +short
dig CAA www.thecollectionisb.com.pk +short
dig CAA com.pk +short

# Dashboard zone
dig CAA thecollectionisb.pk +short
dig CAA admin.thecollectionisb.pk +short
dig CAA pk +short
```

If any exist they must authorise every CA Cloudflare may use — `letsencrypt.org`, `pki.goog`,
`ssl.com`, `digicert.com`, `sectigo.com`, `comodoca.com` — or be removed for the cutover. An incomplete
set blocks issuance exactly as a wrong one does. There is no `cloudflare.com` CAA value; that is a
common and wrong instruction.

### 6. Onboard both zones

Do this **once per domain** — `thecollectionisb.com.pk` and `thecollectionisb.pk`. They are
independent zones with independent nameserver changes, and each has its own Active wait. You can run
them in parallel, but the dashboard cannot be attached until the `.pk` zone reports Active.

For each zone, in this order:

1. **Disable DNSSEC at the registrar first** and wait for it to clear. Changing nameservers with DNSSEC
   active makes the domain hard-unreachable, not soft-degraded.
2. Add the site in Cloudflare, let it import records, then **verify the imported set against the
   registrar's zone by hand** — Cloudflare's scan is best-effort and misses records. Leave the existing
   Vercel records **DNS-only (grey cloud)** so nothing changes behaviourally yet.
3. Lower the TTL on the records that will move to 60s, and wait at least the *old* TTL.
4. Update nameservers at the registrar. Wait for the zone to go **Active** (up to 24 h).
5. Re-enable DNSSEC from Cloudflare's side.

The repo can sit in this dual-config state indefinitely without harm — Vercel stays live throughout.

### 7. Re-issue the API token, now zone-scoped, and set TLS

The token needs **Account → Workers Scripts → Edit**, plus **Zone → Workers Routes → Edit**,
**Zone → DNS → Edit** and **Zone → Zone → Read** scoped to **both zones** — in the token editor's zone
selector, include `thecollectionisb.com.pk` *and* `thecollectionisb.pk` (or select "All zones" on the
account). A single workflow run deploys both Workers, so a token covering only the `.com.pk` zone will
deploy the website and then 403 on the dashboard. Attaching a Custom Domain makes Cloudflare *create a
DNS record*, which the stock Workers template does not clearly permit — without these, step 9 fails.
Update the `CLOUDFLARE_API_TOKEN` GitHub secret.

Then, **for each zone**, under **SSL/TLS → Edge Certificates**: enable **Always Use HTTPS**, and set
the encryption mode to **Full (strict)**. These are per-zone settings, so doing it on `.com.pk` does
nothing for `.pk`. Vercel did http→https redirection automatically; a new Cloudflare zone does not, and
this is otherwise a silent parity regression.

Also enable **HSTS** on both zones — Vercel sent `Strict-Transport-Security` automatically and
Cloudflare does not, so it is the one other Vercel default that would otherwise vanish unnoticed. It
closes the first-request downgrade window for anyone who types a hostname without a scheme, which on
`admin.thecollectionisb.pk` is a request carrying a Supabase session. Set `max-age=86400` for the
48-hour soak, then raise to `max-age=31536000` once cutover is confirmed.

**Leave `includeSubDomains` and `preload` OFF on both.** HSTS is effectively irreversible once a
browser has cached it and unrecoverable once preloaded. This matters more on `thecollectionisb.pk`
than it looks: `includeSubDomains` there would force HTTPS on *every* subdomain of that registration —
including any future mail, staging or vendor host that isn't HTTPS-ready — and `admin` is already
covered on its own without it.

Prefer these zone toggles over a `_headers` rule: `_headers` cannot reach the edge-issued http→https
301, the `www` redirect in step 10, or any zone hostname not served by these Workers.

### 8. Clear the way for `admin.thecollectionisb.pk`

The dashboard hostname is decided: **`admin.thecollectionisb.pk`**, already written into
`apps/dashboard/wrangler.jsonc`. Two things to check before attaching it:

- Whether `admin.thecollectionisb.pk` (or whatever the dashboard answers on today) exists on Vercel,
  so you know what to remove at decommission.
- Whether the imported `.pk` zone already contains **any** record for `admin`. If it does,
  **delete that record** — a Custom Domain cannot attach to a hostname that already has a CNAME, and
  the deploy will fail with a confusing error if you skip this.

**Decided: the public site is served on both domains.** Five hostnames in total across two zones:

| hostname | serves | zone |
| --- | --- | --- |
| `thecollectionisb.com.pk` | website — **canonical origin** | `.com.pk` |
| `www.thecollectionisb.com.pk` | website | `.com.pk` |
| `thecollectionisb.pk` | website (canonicalised to `.com.pk`) | `.pk` |
| `www.thecollectionisb.pk` | website (canonicalised to `.com.pk`) | `.pk` |
| `admin.thecollectionisb.pk` | **dashboard only** | `.pk` |

The dashboard is *only* on `admin.thecollectionisb.pk` — it is a separate Worker with its own route
list, so no website hostname can ever reach it, and `admin` never serves the marketing site.

Serving identical content on two domains splits search signals unless one is declared authoritative,
so the repo neutralises that in two independent ways, both already in place:

1. **`src/app/hooks/use-canonical.ts`** injects `<link rel="canonical">` per route, always on the
   `.com.pk` origin, whichever domain the visitor arrived at. `index.html` deliberately carries **no**
   static canonical — Google's guidance is that if you cannot set it in the HTML source, set it *only*
   via JavaScript rather than emitting one and overriding it.
2. **`public/_headers`** sends `Link: <https://thecollectionisb.com.pk/:splat>; rel="canonical"` on the
   two `.pk` hostnames, covering crawlers that never run JavaScript. `:splat` preserves the path, so
   `/collection/abc` canonicalises to the same path, not the homepage.

Both resolve to the same URL, and `sitemap.xml` lists `.com.pk` URLs only. **Do not add `.pk` entries
to the sitemap, and do not add a static canonical to `index.html`** — either would undo this.

A canonical is used rather than a `noindex` on the `.pk` copy deliberately: `noindex` would drop it
from search and discard any link equity pointing at it, whereas a canonical consolidates that equity
onto `.com.pk`.

### 9. Attach the custom domains from the repo

Uncomment the `routes` blocks in both `wrangler.jsonc` files and push — the real hostnames are already
filled in (`thecollectionisb.com.pk` + `www` for the website, `admin.thecollectionisb.pk` for the
dashboard). The workflow redeploys and Cloudflare creates the DNS records and issues certificates
itself.

**Both zones must read Active first.** If only `.com.pk` is Active, uncomment the website's routes
alone and push; come back for the dashboard when `.pk` lands. A route pointing at a non-Active zone
fails the deploy.

**Do not hand-create any DNS record for these hostnames first.** Custom Domains match exact hostnames
with no wildcards, so the apex and `www` are two separate entries.

### 10. Optional — collapse the `www` hostnames

As configured, all four website hostnames **serve** the site, with the `.pk` pair canonicalised to
`.com.pk`. That is a complete, correct setup and needs nothing further.

If you would rather `www` 301 to its bare apex on either domain: drop that `www` route from
`apps/website/wrangler.jsonc`, create a **proxied (orange-cloud)** placeholder `A` record for `www`
→ `192.0.2.1` in that zone, and add a Redirect Rule to the apex with 301, preserve-query-string and
subpath matching. The canonical arrangement is unaffected either way.

### 11. Re-verify, soak, then decommission

Re-run the whole of step 4 against the **real domains**, plus:

Re-point the step 4 variables at the real hostnames first — `W=https://thecollectionisb.com.pk` and
`D=https://admin.thecollectionisb.pk` — then re-run that whole block, plus these per-zone checks:

```bash
# http -> https on BOTH zones (per-zone setting; passing on one proves nothing about the other)
curl -sI -o /dev/null -w '%{http_code} %{redirect_url}\n' http://thecollectionisb.com.pk/
curl -sI -o /dev/null -w '%{http_code} %{redirect_url}\n' http://admin.thecollectionisb.pk/

# HSTS present on both
curl -sI https://thecollectionisb.com.pk/ | grep -i strict-transport
curl -sI https://admin.thecollectionisb.pk/ | grep -i strict-transport

# The public site must NOT be de-indexed on its real domain...
curl -sI https://thecollectionisb.com.pk/ | grep -i x-robots-tag \
  && echo "PRESENT — WRONG, the public site is de-indexed"

# ...and the dashboard MUST be, on every path.
for p in / /anything /favicon.svg; do
  curl -sI "https://admin.thecollectionisb.pk$p" | grep -qi x-robots-tag || echo "MISSING on $p"
done

# All four website hostnames must serve the site
for h in thecollectionisb.com.pk www.thecollectionisb.com.pk thecollectionisb.pk www.thecollectionisb.pk; do
  curl -sI -o /dev/null -w "$h %{http_code}\n" "https://$h/"
done

# The .pk copies must declare .com.pk canonical, PATH-PRESERVING (JS-free half).
curl -sI https://thecollectionisb.pk/collection | grep -i '^link:'
# expect: Link: <https://thecollectionisb.com.pk/collection>; rel="canonical"

# The .com.pk origin must NOT send a Link canonical (it is the canonical).
curl -sI https://thecollectionisb.com.pk/collection | grep -i '^link:' \
  && echo "UNEXPECTED — the canonical origin should not canonicalise elsewhere"

# The JS half: rendered canonical must be .com.pk on BOTH domains, per route.
# Load https://thecollectionisb.pk/collection in a browser and confirm
# document.querySelector('link[rel=canonical]').href is the .com.pk URL.

# admin must serve the DASHBOARD, never the marketing site.
curl -s https://admin.thecollectionisb.pk/ | grep -q "Operator dashboard" \
  && echo "admin serves the dashboard" \
  || echo "CHECK — admin is not serving the dashboard shell"
```

Confirm `robots.txt`'s `Sitemap:` line and the four `<loc>` entries in `sitemap.xml` still match the
live domain.

**Done.** Vercel was decommissioned once the Cloudflare deployment was verified live on both domains
(2026-07-23): the domains were removed from the Vercel projects, the projects deleted, and the two
`vercel.json` files plus the `.vercel/` ignore entry removed from the repo. Cloudflare Workers is now
the sole host; the sections above are retained as the record of how the cutover was performed.

> Now that Vercel is gone, adding a `packageManager` field (`"pnpm@11.0.8"`) to the root
> `package.json` is safe — it was withheld only because Vercel read it and would have switched to a
> Corepack pnpm version its build image did not ship, breaking the rollback target. It is optional:
> `pnpm/action-setup` in CI and the local toolchain are already pinned without it.

### 12. Harden the dashboard

- Set `"workers_dev": false` in `apps/dashboard/wrangler.jsonc` and push — this removes the public
  workers.dev surface. The noindex header stops crawlers; it does not gate humans.
- Strongly consider **Cloudflare Access** with an Allow policy naming the specific operator emails.
  Check the seat allowance and price in your own billing UI rather than assuming it is free.

## Rollback

Vercel is gone, so every rollback path is now within Cloudflare:

- **Bad deploy:** `npx wrangler@4.113.0 rollback --message "reason"` from the app directory (`cd
  apps/<app>`), which reverts that Worker to its previous version, or roll back from the dashboard:
  Workers & Pages → the Worker → Deployments → an earlier version → **Rollback**.
- **Bad commit:** revert it on `master` and let Cloudflare Workers Builds redeploy, or run the manual
  fallback workflow (**Actions → Deploy to Cloudflare Workers → Run workflow**) for a single app.
- **Removing a Custom Domain does not delete its certificate.** Clean it up manually in SSL/TLS → Edge
  Certificates.

## Deploying manually

```bash
pnpm --filter @collection/website build     # with VITE_* in the environment
cd apps/website && npx --yes wrangler@4.113.0 deploy
```

`wrangler` is deliberately not a workspace dependency: it pulls in `workerd`, whose install script
would trip pnpm's `strictDepBuilds` and hard-fail `pnpm install`. Invoking it through `npx` keeps it
out of the pnpm graph and leaves the lockfile untouched.
