# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

Two front-end apps for **The Collection**, a luxury-car dealership in Islamabad, sharing one
Supabase project and one brand:

- **Dashboard** — an operator tool: an AI-assisted social inbox + light CRM (customers message
  over Instagram / WhatsApp / Messenger, a bot handles routine replies, staff take over), plus
  tickets, viewing appointments, buyer **Matching**, and multi-showroom **Inventory**.
- **Website** — the public, by-appointment marketing + collection site. Read-only; every CTA is a
  WhatsApp deep-link.

Both originated as **Figma Make** exports, so some conventions are dictated by that toolchain
(see Vite section).

**What is real vs. mock.** Supabase now backs **inventory, showrooms, auth, notifications,
enquiries/matching, and car-image storage**. Still **in-memory seed data** from
`apps/dashboard/src/app/data/mock.ts`: **conversations, tickets, appointments, customers** — i.e.
the whole social-inbox half, pending the bot session. Treat `mock.ts` as authoritative for those
four entities and `@collection/shared` as authoritative for everything else.

## Repository layout (monorepo)

A **pnpm workspace** with two apps and one shared package:

- `apps/dashboard/` — the operator dashboard (`@collection/dashboard`). **Paths below are relative
  to `apps/dashboard/`** unless stated otherwise.
- `apps/website/` — the public site (`@collection/website`). The only app that uses `react-router`
  (`/`, `/collection`, `/collection/:id`, `/privacy`, `/terms`, `*`).
- `packages/shared/` — `@collection/shared`: the Supabase data layer **both apps import**, plus the
  brand design tokens at `packages/shared/styles/tokens.css`.
- `supabase/` — SQL migrations, Edge Functions (`functions/`), + `PROVISIONING.md` (the
  account-creation runbook). Repo root.
- `brand/` — the standalone brand sheet (`brand-sheet.html`) and logo artwork. Repo root.

Each app is a self-contained Vite project with its own `package.json`, `vite.config.ts`,
`index.html`, `wrangler.jsonc`, and `.env` (see each app's `.env.example`; both point at the **same**
Supabase project).

Ignore `apps/*/README.md` and `apps/*/guidelines/Guidelines.md` — unedited Figma Make boilerplate.

## Commands

**pnpm-only** — the root `pnpm-lock.yaml` is the single lockfile.

```bash
pnpm install                              # install all workspace deps (run at the repo root)
pnpm --filter @collection/dashboard dev   # dashboard dev server (or: cd apps/dashboard && pnpm dev)
pnpm --filter @collection/dashboard build # dashboard production build
pnpm --filter @collection/website  dev    # website dev server
pnpm --filter @collection/website  build  # website production build
```

There is **no linter and no typecheck** — and no `tsconfig.json` or `typescript` dependency
anywhere. `vite build` transpiles via esbuild **without type checking**, so a type error fails
nothing; it only shows in an editor. Do not invent lint/typecheck commands.

Treat a green build as very weak evidence. It catches syntax errors and nothing else — a temporal
dead zone `ReferenceError` that white-screened the whole dashboard, and a filter that hid the cars
it was meant to show, both compiled cleanly.

There is **one** test, and no test runner:

```bash
pnpm --filter @collection/dashboard test:filters   # node --experimental-strip-types, no deps
```

It asserts the inventory filter logic in `components/inventory/car-filter.ts`, which is kept free
of React and Radix imports precisely so it can be run this way. Do not assume any other test
command exists.

`pnpm-workspace.yaml` carries three deliberate policies: `allowBuilds` (only esbuild and
`@tailwindcss/oxide` may run install scripts), `minimumReleaseAge: 10080` (refuse packages published
less than 7 days ago), and `supportedArchitectures` pinned to linux — which is why the **root**
`package.json` lists the win32 native binaries (`@rollup/rollup-win32-x64-msvc`,
`@tailwindcss/oxide-win32-x64-msvc`, `lightningcss-win32-x64-msvc`) as `optionalDependencies` so
local Windows dev works.

## Deployment (Cloudflare Workers Static Assets)

Both apps deploy to **Cloudflare Workers Static Assets** — two assets-only Workers,
`collection-website` and `collection-dashboard`, each configured by `apps/<app>/wrangler.jsonc`.
`.github/workflows/deploy.yml` builds and deploys both on a push to `master`. **`DEPLOYMENT.md` at
the repo root is the full runbook** (cutover sequence, verification commands, rollback).

- **Assets-only is deliberate: neither config sets `main`.** Cloudflare does not apply `_headers`
  rules to responses generated by Worker code, so introducing a Worker script would silently drop
  the dashboard's `X-Robots-Tag`. It also keeps every request free and unmetered.
- **SPA routing** is `assets.not_found_handling: "single-page-application"` — unmatched paths get
  `/index.html` with a 200, while real files in `dist/` still win because asset matching runs first.
  Do **not** replace this with a `_redirects` `/*` rule: redirects are followed regardless of
  whether an asset matches, which would shadow `robots.txt`, `sitemap.xml` and `og-image.jpg`.
- **Response headers live in `apps/<app>/public/_headers`**, not in `wrangler.jsonc` (which has no
  `headers` key). They only work from inside `assets.directory`, and reach it via Vite's verbatim
  copy of `public/` → `dist/`. CI asserts `dist/_headers` exists on every build. The dashboard's
  noindex header is the third layer of its never-index guarantee, alongside `robots.txt` and the
  `<meta name="robots">` tag.
- **`VITE_SUPABASE_URL` / `VITE_SUPABASE_ANON_KEY` are BUILD-TIME only.** Vite inlines them into the
  bundle. They are GitHub Actions repository secrets consumed by the build step. Setting them as
  Cloudflare Worker runtime vars or `wrangler secret put` would do nothing — a different store, and
  there is no Worker script to read them from. The symptom is `undefined` in the bundle with no
  build error.
- **CI builds, not Cloudflare.** Cloudflare's build image ships pnpm 10.11.1, which predates this
  repo's `allowBuilds` (needs ≥ 10.26.0) and `minimumReleaseAge` (≥ 10.16.0) and silently ignores
  them — so `esbuild` and `@tailwindcss/oxide` would never run their install scripts. The workflow
  pins pnpm 11.0.8 and reads Node from `.node-version`. Keep `supportedArchitectures.os: [linux]` in
  `pnpm-workspace.yaml`: it is what puts the Linux native binaries in the lockfile.
- `wrangler` is intentionally **not** a workspace dependency — it pulls in `workerd`, whose install
  script would trip pnpm's `strictDepBuilds`. CI invokes it via a pinned `npx wrangler@<version>`.

**The website is served on two domains; the dashboard on one.** `thecollectionisb.com.pk` (apex +
`www`) and `thecollectionisb.pk` (apex + `www`) both serve the public site from the one Worker; the
dashboard is only ever on `admin.thecollectionisb.pk`. `.com.pk` is the **canonical** origin, and
three things must agree on that or the two copies compete in search:
`src/app/hooks/use-canonical.ts` (injects `<link rel="canonical">` per route), the `Link: … rel="canonical"`
rules for the `.pk` hostnames in `apps/website/public/_headers`, and the `<loc>` entries in
`sitemap.xml` plus the `Sitemap:` line in `robots.txt`. **Never add a static `<link rel="canonical">`
to `apps/website/index.html`** — one HTML file serves every route, so it would canonicalise them all
to the same URL, and Google's guidance is to set it only via JS when the source cannot carry it.

## The shared data layer (`@collection/shared`)

Consumed as **TypeScript source** — no build step, no `dist`. Each app's Vite compiles it, which is
what lets `import.meta.env.VITE_*` resolve **per app**. `index.ts` re-exports everything:

| Module | Owns |
| --- | --- |
| `car.ts` | The `Car` type + `CarStatus`. Single source of truth for a car's shape. |
| `supabase.ts` | The one `supabase` client (anon key, from `VITE_SUPABASE_URL` / `VITE_SUPABASE_ANON_KEY`). |
| `auth.ts` | `signIn`/`signOut`/`getSession`/`onAuthStateChange` + `authContextFromSession`. |
| `showrooms.ts` | `Showroom`, `listShowrooms`, `getShowroomById`, `MASTER_SHOWROOM_ID`. |
| `inventory.ts` | Every read/write of the `inventory` table + `subscribeInventory`. |
| `storage.ts` | Car-photo upload/delete against the `car-images` bucket. |
| `notifications.ts` | Admin-only "a partner added a car" feed + `subscribeNotifications`. |
| `enquiries.ts` | Buyer enquiries, the `match_enquiry` RPC, `subscribeEnquiries`. |
| `partners.ts` | Admin-only partner provisioning. **The one module that calls an Edge Function, not a table** — see below. |

**This package is the only place that knows database column names.** The DB is `snake_case`
(`mileage`, `docs_complete`, `showroom_id`); the app is `camelCase` (`mileageKm`, `docsComplete`,
`showroomId`). Every module keeps a private `…Row` interface plus `rowTo…` / `…ToRow` mappers.
Add a column → change the mapper here, never in a component. The comment in `inventory.ts` states
the intent: the LLM bot will `import { listAvailableCars } from "@collection/shared"` without
touching React.

## Auth, roles and tenancy

Real Supabase Auth. **Role and showroom come from the user's `app_metadata` JWT claim** — server-set
and not user-editable. Never read `user_metadata` for authorization (a user can edit their own and
would self-escalate).

**Partner accounts are provisioned from the dashboard** (Partners screen → the `admin-partners` Edge
Function). **Admin and photographer accounts are still stamped by hand** in the Supabase SQL editor —
see `supabase/PROVISIONING.md`. There is no invite flow for anyone.

| Role | Scope |
| --- | --- |
| `admin` | Everything, all showrooms. The only role that sees Conversations / Tickets / Appointments / Matching / notifications. |
| `partner` | Their own showroom's inventory only. |
| `photographer` | Inventory **content** on The Collection's own showroom: add cars, edit photos/specs. No delete, no status change, no publish/feature. |

A **showroom is a tenant**; every car belongs to exactly one. The Collection is the fixed master
showroom `MASTER_SHOWROOM_ID` (`11111111-…`) — the only showroom whose cars can appear on the public
website.

**Two independent per-partner visibility flags** on `showrooms`, both admin-toggled from the Partners
screen, both defaulting to false:

| flag | migration | lets that partner also SEE |
| --- | --- | --- |
| `can_view_master` | 0020 | The Collection's (master) inventory |
| `can_view_partners` | 0021 | every *other* partner's inventory |

They compose, so a partner sees their own showroom plus whichever axes are on. Both widen **SELECT
only** — INSERT/UPDATE/DELETE still read `admin OR showroom_id = app_showroom_id()`, so a flagged
partner can never edit, publish, feature or delete a car they don't own. 0021 additionally widens the
**showrooms** SELECT policy, mirroring the inventory branches exactly, so the UI can name whose car it
is showing; a caller may read a showroom row iff they may already read that showroom's cars.

`can_view_partners` is **one-directional** — granting it to A does not let B see A — and the partner
being viewed does not consent, which is why it is off by default and enabled one partner at a time.
Per-pair control ("A may see B but not C") would need a junction table; the boolean is the wrong
shape for it.

Both are **columns, not JWT claims**, deliberately: a claim would not take effect until the partner
signed out and back in. Client-side, a partner with *either* flag must fetch **unscoped**
(`fetchShowroomId = undefined`) and let RLS decide what comes back — asking for one `showroom_id`
would filter the very rows the grant exists to reveal straight back out. `App.tsx` calls this
`seesForeignCars`, and it also drives `editableShowroomId`, which makes Inventory treat any car
outside the partner's own showroom as read-only.

`App.tsx` derives `isAdmin` / `isPhotographer` / `myShowroomId` from the session and gates the UI:
a non-admin gets `navList` filtered to Inventory and `shownView` forced to `"inventory"`. An
allowlist (`configured`) rejects any account that isn't admin-or-(partner/photographer-with-showroom),
so a mis-stamped claim can never fall through to the all-showrooms fetch path.

**The UI flags are convenience; RLS in `supabase/migrations` is the real boundary.** Every permission
the UI hides is independently enforced in the database (0006 inventory lockdown, 0007 notifications,
0009 storage, 0010 enquiries, 0014 publish guard, 0015 public-website scope, 0019 photographer).
When adding a capability, add the policy **and** the UI flag.

Realtime honours RLS, so subscriptions are self-scoping: a partner's stream carries only their own
rows. Three feeds are live — `subscribeInventory`, `subscribeNotifications`, `subscribeEnquiries` —
and each returns an unsubscribe function that **must** be called on unmount.

## Dashboard architecture

**All application state lives in `apps/dashboard/src/app/App.tsx`.** One component holds every entity
list in `useState`, all mutation handlers, all data fetching, and cross-screen navigation. There is
**no router** (despite `react-router` being a dependency) and **no global store / context** — `view`
is a `useState<ViewKey>` and everything is **prop-drilled** into screens. To add behavior: add a
handler in `App.tsx` and thread it through props.

The mutation pattern for Supabase-backed entities is consistent — *call the shared layer → re-fetch →
toast*. Do not optimistically patch local state for inventory or enquiries; `loadCars({silent:true})`
/ `loadEnquiries()` re-pull so the UI reflects what the DB actually stored (server-set fields, the
full `photos` array). Notification read-marking is the one exception: it patches locally first
because realtime reconciles it.

Layout shell wraps the active screen: `Sidebar` (desktop) / `BottomNav` (mobile) + `TopBar`
(global search, showroom identity, notifications, avatar menu), with `CustomerProfileSheet`,
`CarDetailSheet` and `NotificationsSheet` as app-level overlays.

Directory map under `apps/dashboard/src/app/`:
- `data/mock.ts` — the **mock half** of the domain model: types for `Conversation`, `Ticket`,
  `Appointment`, `Customer`, `Message`, `Channel` and their status unions, the seed arrays, and the
  shared formatters/derivations (`formatCurrency`, `relativeAge`, `ticketAgeBucket`,
  `windowClosingSoon`). Car types come from `@collection/shared`, not here.
- `components/screens/` — one component per `ViewKey` (Home, Conversations, Tickets, Appointments,
  Matching, Inventory, Partners) + SignIn. The `ViewKey` union and nav list live in
  `components/shell/nav-items.ts`; an item marked `desktopOnly` (Partners) is kept out of the mobile
  `BottomNav`, which is sized for six tabs.
- `components/shell/` — Sidebar, BottomNav, TopBar, GlobalSearch, NotificationsSheet, AvatarMenu.
- `components/{inventory,matching,partners,conversation,customer,appointments,brand,common}/` —
  feature widgets. `common/` holds StatusPill, ChannelBadge, AgingDot, SectionHeader.
- `components/ui/` — **shadcn/ui (Radix-based)**. Generated; treat as a vendored library.
  `ui/utils.ts` exports the `cn()` class-merge helper used everywhere.
- `components/figma/ImageWithFallback.tsx` — use for remote images.

**Seed↔live id mismatch:** seed conversations and appointments reference mock car ids (`car1`…) while
live inventory has Supabase UUIDs. `remapSeedCarIds` in `App.tsx` rewrites those refs once on first
load by matching make+model. It disappears when conversations move to the database.

### Matching

Buyer enquiries only — a car offered for sale is added to inventory instead (migrations 0016-0018
retired selling). An enquiry is a wishlist: `year` = minimum, `mileageMaxKm` = maximum, `price` =
budget, `docsComplete: true` = requires full docs. Enquiries **expire after one month**
(`isEnquiryActive` = status active AND not past `expiresAt`); `renewEnquiry` resets the window and
day-clamps exactly like Postgres interval math.

Matching is **rules-based, on-demand, and server-side**: the `match_enquiry` RPC is `SECURITY
INVOKER`, so the admin's all-showroom RLS lets it match a buyer against any showroom's stock, tiered
`exact` / `possible`. The reverse push is a DB trigger (0011): adding a car that matches an active
buying enquiry notifies the admin — no app code involved.

### Partners (the one server-side surface)

`supabase/functions/admin-partners/` is the repo's **only Edge Function**, and the only place the
**service-role key** exists. It exists because two things are impossible from the browser with the
anon key: creating an auth user with a password, and stamping the trusted `app_metadata` claim. The
Partners screen (admin-only) calls it through `packages/shared/partners.ts` to create, remove, reset
the password of, and set visibility on partner showrooms.

The security boundary is **inside the function**, not RLS — it runs as service role, so it
re-verifies the caller's JWT and requires `app_metadata.role === 'admin'` on *every* action. If you
add an action, add that check with it. Passwords are never logged or returned.

Create rolls back: if the auth user can't be made, the showroom row it just inserted is deleted, so a
tenant with no way in can't exist. Remove refuses while the showroom still holds cars
(`inventory.showroom_id` is `ON DELETE RESTRICT`) and only proceeds with an explicit `force`, which
also deletes those cars and their storage objects.

Deploy it with `supabase functions deploy admin-partners` — it is **not** part of the Cloudflare
build, and a dashboard deploy does not update it.

## Website

Read-only. `useInventory()` loads published Collection cars once with a 12s timeout → calm error
state. The client-side `showroomId: MASTER_SHOWROOM_ID` scope is belt-and-braces on top of the RLS in
0015; keep both. Status is deliberately **not** filtered — available, reserved and sold all render,
ordered by `byDisplayOrder`.

`published` / `featured` are **admin-only** per-car booleans (0014) that drive the public site:
`published` → the `/collection` listing, `featured` → the landing page strip (featured implies
published). A DB trigger rejects a non-admin write to either, which is why `carToRow` emits those
columns **only when the caller supplies them** and the Inventory form `delete`s them from the payload
when `!canPublish`. Preserve that: a non-admin edit must omit the keys entirely, not send `false`.

`apps/website/src/config/contact.ts` is the single source for every phone number, WhatsApp
deep-link, social URL and message template. Prices use the crore/lakh convention (`formatPKR`), not
`mock.ts`'s `formatCurrency`.

## Styling & design system

Tailwind **v4** (config-less, via `@tailwindcss/vite`).

**Brand colours live in ONE file: `packages/shared/styles/tokens.css`.** Both apps `@import` it at
the top of their own `src/styles/theme.css`. **Never redefine a brand colour in an app** — change it
in the shared file and both apps follow. Each app's `theme.css` keeps only what is genuinely
app-specific (surface polarity, the shadcn semantic layer, signals, radii) and re-exports tokens to
Tailwind via `@theme inline`, which is what mints the utilities (`bg-noir`, `text-ink-60`, …).
Tailwind's own defaults (`--color-white` / `--color-black`) are retargeted onto tokens there too, so
`bg-white` follows the palette instead of being stranded on a literal.

Colour rules:
- Use the token utilities (`bg-noir`, `text-accent`, `border-card-border`), and the **opacity
  modifier** for alpha (`bg-accent/20`, `text-cream/74`) — never a hand-written `rgba()`.
- In a CSS value (inline `style`, a gradient), use `color-mix(in srgb, var(--token) N%, transparent)`.
- Never a hard-coded hex.

The palette is **monochrome + silver — there is no gold.** Neutral blacks (`noir` #0A0A0A,
`graphite` #141414), warm whites (`alabaster`/`cream` #F4F2ED, `fog` #A7A29B), and a cool silver
accent. Plus `platinum-soft`, `ch-instagram/whatsapp/messenger`, and `signal-amber/red/good`.

**The accent is `--accent`, and it resolves PER APP** — same silver, two weights:
- **Dashboard** (light work surfaces): `--accent` = **steel** `#6F7276` (4.83:1 AA on white). Its
  dark chrome (sidebar, bottom nav, avatar menu, sign-in) uses `--accent-on-dark` = **platinum**
  `#D6D8DB`, because steel is only 4.10:1 on noir and fails AA as text.
- **Website** (fully dark): `--accent` = **platinum** `#D6D8DB`. **Never put platinum on a light
  surface — 1.43:1.**

The website also addresses **role tokens**, never raw brand tokens, for surfaces and text
(`--surface-page` / `--surface-raised` / `--surface-deep`, `--text-primary` / `--text-body` /
`--text-muted`). They exist because `--noir` had to be both a background and the primary text before
the inversion — one value could not serve both.

The website owns the metal gradient; the dashboard uses flat steel and no gradient. Two cuts, and
they are not interchangeable: `.metal-fill` (CTAs) takes `--metal`, while `.text-metal` takes
`--metal-text`, a re-cut whose darkest stop clears AA as **small** text (7.99:1) — so `.text-metal`
is safe at any size, including 16px card prices. Also website-only: `.hairline`, `.film-grain`,
`.sheen`.

Signal colours are tuned per surface (a signal that passes on a white card fails on a black one), so
`--signal-*` values live in each app's `theme.css`, not in the shared file.

Typefaces are named for what they ARE: `--typeface-sans` (Inter) and `--typeface-serif` (Fraunces),
self-hosted via `@fontsource` and imported in `main.tsx` — no runtime Google Fonts request. (There
used to be a `--font-display` that meant Inter in the dashboard and Fraunces on the website — same
name, two typefaces. Don't reintroduce it.)

Custom base classes (dashboard `theme.css`): `.eyebrow`, `.editorial`, `.editorial-italic`,
`.script`, `.mark`, `.accent-rule`. `src/styles/index.css` is each app's entry (imported by
`main.tsx`); `globals.css` is intentionally empty in both apps.

## Vite specifics (Figma Make constraints)

- In each app, `@` aliases to that app's `src/` (`vite.config.ts`, resolved via `__dirname`).
- A custom `figma:asset/...` import resolver maps to the app's `src/assets/`.
- `assetsInclude` covers `.svg` and `.csv` for raw imports — **never add `.css`/`.ts`/`.tsx`** there.
- The React and Tailwind Vite plugins are both required by Make even where unused — do not remove them.

## Conventions

- Mock entities are referenced across screens by id (`customerId`, `carId`, `conversationId`); resolve
  to objects by `.find()` against the lists passed from `App.tsx`.
- All timestamps are ISO strings; new ones are `new Date().toISOString()`. Ids for **mock** entities
  are `` `prefix${Date.now()}` `` (e.g. `reply`); Supabase-backed entities get server-generated UUIDs —
  never mint an id client-side for those.
- Display text for a DB enum belongs in one module, not at the call site — see
  `components/matching/channel.ts` (`walk_in` → "Walk-In").
- Migrations are append-only and numbered `NNNN_name.sql`; each opens with a comment block explaining
  the why. Follow that when adding one, and note whether it is destructive.
- MUI (`@mui/material`) is installed but the UI is built almost entirely on the shadcn `ui/` layer —
  prefer those components for consistency.
